# ManJuFlow｜漫剧流 第一阶段 MVP 演示

## 演示目标

本次演示的目标，是向老板说明 ManJuFlow｜漫剧流当前完成的是第一阶段文本链路 MVP，不是最终的自动成片系统。

当前版本重点验证从“灵感输入”到“结构化短剧剧本输出”的基础链路是否清晰、可用、可继续扩展。文生图、文生视频、自动剪辑、完整生产流水线会在后续阶段逐步接入。

## 当前已完成能力

- 本地后端服务
- 健康检查接口
- 结构化数据 Schema
- 灵感转剧本 mock 接口
- 前端演示页
- 前端调用后端展示结构化剧本
- 本地 Git 版本管理

## 演示步骤

1. 启动后端：

```bash
cd apps/api
uvicorn app.main:app --reload
```

2. 启动前端：

```bash
cd apps/web
npm run dev
```

3. 打开前端页面：

```text
http://localhost:5173/
```

4. 在页面输入一个灵感，例如：

```text
一个被裁员的中年男人，意外发现公司老板用 AI 伪造财报
```

5. 点击“生成结构化剧本”。

6. 展示生成结果，包括：

- 项目标题
- 一句话卖点
- 角色列表
- 分集大纲
- 场景内容
- 对白

7. 点击“复制 JSON”，说明当前结构化结果可以被后续系统继续消费。

## 讲解话术

### 为什么先做文本链路

AI 影视化创作的上游核心是文本结构。只有先把灵感、剧本、角色、分集、场景、对白这些信息稳定结构化，后续分镜、绘图 Prompt、视频生成和剪辑流程才有可靠输入。

所以第一阶段先验证“灵感到结构化剧本”的闭环，而不是直接进入文生图或文生视频。

### 为什么先 mock 再接真实 LLM

当前 mock 接口用于先稳定产品形态、接口契约和前后端联调流程。这样可以先确认页面、字段、数据结构和业务链路是否合理。

等结构稳定后，再把 mock 层替换为真实 LLM 调用，可以降低返工风险，也便于后续比较不同模型和 Prompt 的效果。

### 为什么模块化拆分

项目按后端、前端、Schema、接口文档、运行文档分开推进。这样做的好处是边界清晰，每一部分都可以单独验证和迭代。

后续接入 Prompt、LLM、历史记录、导出、分镜、绘图 Prompt 时，也可以在现有结构上逐步扩展，而不是一次性堆成一个难维护的系统。

### 为什么本地验证后再部署服务器

当前仍处于第一阶段 MVP，本地验证可以更快发现接口、页面和数据结构问题，避免过早投入部署和环境维护成本。

在本地链路稳定后，再部署到服务器会更稳妥，也更容易判断部署问题和业务问题分别在哪里。

### 后续如何扩展到分镜、Prompt、文生图、文生视频

后续会在结构化剧本的基础上继续拆解：

- 从剧本扩展到分镜结构
- 从分镜生成绘图 Prompt
- 从绘图 Prompt 接入文生图工具
- 从关键画面扩展到文生视频
- 最后再考虑自动剪辑和完整流水线编排

当前版本的价值，是先把后续所有生成环节需要依赖的文本输入层打稳。
