# Boss Demo Script｜老板演示脚本

## 1. 演示目标

本演示不是为了展示“炫技功能”，而是证明 ManJuFlow 可以成为面向编剧、短剧策划、漫剧内容团队的 AI 短剧剧本生成与改编工作台。

演示重点：

- 内部账号 / mock login 后进入工作台；
- 三入口选择；
- 灵感生成短剧剧本；
- 电影剧本改编短剧剧本；
- 小说改编短剧剧本；
- AI Assistant 未来辅助改写、改编、增强钩子；
- 在线编辑、下载 / 导出、上传作为市场试用关键能力；
- 切分 / 分镜 / Prompt 是下一大功能预备。

演示时应把产品价值说清楚：ManJuFlow 当前不是优先做自动成片，也不是优先做文生图 / 文生视频，而是先解决编剧行业最核心的短剧文字内容生产与改编问题。

## 4. 开场话术

可以直接对老板这样说明：

> 这个项目当前不是为了马上做自动成片，也不是先追文生图或文生视频。现在更有价值的是先把编剧行业最刚需的短剧文字生产做好：从一个灵感、一个电影剧本片段，或者一段小说文本，快速生成可编辑、可导出、可继续进入后续分镜和 Prompt 链路的短剧剧本。

## 5. 演示路径 A：灵感生成短剧剧本

适用场景：从 0 到 1 的原创短剧内容开发。

演示步骤：

1. 打开网页；
2. 登录 / mock login；
3. 选择“灵感生成短剧”；
4. 输入一个安全虚构灵感；
5. 选择集数；
6. 生成短剧剧本；
7. 展示分集、人物、钩子、场景；
8. 展示字数限制；
9. 展示复制 / 导出；
10. 说明后续可进入切分 / 分镜 / Prompt。

安全虚构示例灵感：

```text
一位年轻编剧在深夜地铁站捡到一本没有署名的旧剧本。每当她改写其中一幕，现实中的某个陌生人就会收到一段改变命运的消息。她必须在天亮前找出剧本真正的主人。
```

讲解要点：

- 灵感可以很粗糙；
- 系统会把灵感转成短剧剧本；
- 集数选择必须生效；
- 结果应能看到人物、分集、钩子和场景；
- 这不是单次聊天，而是可继续编辑、导出、传递到下一阶段的工作流。

## 6. 演示路径 B：电影剧本改编短剧

适用场景：已有电影剧本、长剧本、影视分场文本，希望改成短剧节奏。

当前说明：

- 如果电影剧本改短剧功能尚未完整落地，应明确标注为规划 / mock 路线；
- 不要把规划能力说成已经完全上线；
- 演示素材必须使用安全虚构文本。

演示步骤：

1. 选择“电影剧本改短剧”入口；
2. 粘贴虚构电影剧本片段；
3. 选择目标集数和短剧风格；
4. 说明输出会强调删减策略、短剧钩子、分集节奏；
5. 展示规划中的输出结构，例如改编策略、保留剧情、分集大纲和每集钩子；
6. 说明后续真实 LLM 接入后会使用独立 prompt，不和灵感生成混用。

安全虚构电影剧本片段：

```text
【片段】一名过气女演员回到废弃片场，准备拍完父亲生前未完成的最后一个镜头。她发现片场里每个道具都对应着父亲当年失踪案的一条线索。制片人劝她离开，但她坚持开机。午夜，摄影机自动亮起，监视器里出现了父亲二十年前留下的影像。
```

讲解要点：

- 电影剧本改短剧不是简单压缩；
- 系统需要识别核心人物、主线冲突和必须保留的剧情；
- 输出应体现短剧节奏、强钩子和分集推进；
- 版权和改编权需要用户自行确认。

## 7. 演示路径 C：小说改编短剧

适用场景：已有小说、网文、故事文本，希望改成短剧剧本。

当前说明：

- 如果小说改短剧功能尚未完整落地，应明确标注为规划 / mock 路线；
- 不要把规划能力说成已经完全上线；
- 演示素材必须使用安全虚构文本。

演示步骤：

1. 选择“小说改短剧”入口；
2. 粘贴虚构小说片段；
3. 选择目标集数、人物关系、改编风格；
4. 说明输出会把叙事文本改成场景、对白、冲突和分集钩子；
5. 展示规划中的输出结构，例如人物关系、主线冲突、分集钩子和场景化改写；
6. 说明长小说需要 chunking，不能一次性全部丢给模型。

安全虚构小说片段：

```text
雨停后的清晨，沈南星在旧书店的二楼发现一本被水泡皱的日记。日记里记录着她母亲年轻时参与的一场舞台事故。更奇怪的是，最后一页写着她今天才会做出的选择：如果你打开这扇门，就不要再相信任何掌声。
```

讲解要点：

- 小说改短剧要把叙事改成可拍的场景；
- 系统需要提炼人物关系、情绪变化和主线冲突；
- 输出应更像短剧剧本，而不是小说摘要；
- 长文本需要字数限制和分块策略。

## 10. 演示路径 E：AI Assistant 规划说明

Assistant 是编剧助手 / 改编助手 / 工作流助手，不是普通聊天框。

Assistant 未来可以：

- 帮用户改写灵感；
- 帮用户提炼电影剧本改编策略；
- 帮用户梳理小说人物关系；
- 帮用户增强短剧钩子；
- 帮用户把结果带入下一步；
- 通过 suggested_actions 执行，但需要用户确认。

边界：

- Assistant 会独立 schema / service / endpoint / prompt；
- 即使使用 DeepSeek，也会与三入口主生成链路隔离；
- 所有 suggested actions 都需要用户确认；
- Assistant 不应自动修改生产数据；
- Assistant 不应跨项目读取上下文。

## 11. 老板关心问题回答

### Q：现在项目主打什么？

A：AI 短剧剧本生成与改编工作台。当前主入口是三类：灵感生成短剧剧本、电影剧本改短剧、小说改短剧。

### Q：现在能不能直接做市场？

A：可以进入受控小范围试用 / 演示验证，但还不是生产级 SaaS。当前适合验证编剧行业对“AI 短剧剧本生成与改编工作台”的需求。

### Q：为什么不先做文生图 / 文生视频？

A：当前老板和用户重点是编剧行业文字内容生产。先把剧本生成、改编、编辑、导出、上传和 Assistant 做稳，后续接图片、视频或 ComfyUI 会更稳，也更容易控制成本、安全和交付质量。

### Q：电影剧本 / 小说改编是否有版权风险？

A：有。需要用户确认其拥有合法使用和改编权。ManJuFlow 只能作为技术 / 商务边界提示，不冒充法律结论，也不承诺法律意义上的版权合规判断。

### Q：AI Assistant 是普通聊天吗？

A：不是。Assistant 是编剧助手 / 改编助手 / 工作流助手，辅助三入口生成、改写、增强钩子和带入下一步。所有 suggested actions 都需要用户确认。

### Q：切分 / 分镜 / Prompt 去哪里了？

A：保留为生成短剧剧本后的下一大功能预备。当前先把短剧剧本生成和改编作为主入口做好，再让结果进入后续切分、分镜和 Prompt。

### Q：用户上传剧本放哪里？

A：当前 mock 阶段不保存真实文件；真实上传第一版会使用受控本地目录并进入 `.gitignore`，后续用 SQLite 记录 metadata。市场试用前会明确清理策略和隐私提示。

### Q：公开仓库安全吗？

A：公开仓库只放工程主干、mock 能力、文档和安全样例，不放 API Key、真实客户数据、真实服务器、模型权重、私有 workflow。

### Q：还有什么没完成？

A：三入口 mock / LLM 链路、电影剧本改短剧、小说改短剧、Assistant UI、真实上传、在线编辑、Word 下载上传、基础门禁和市场试用总结仍需继续推进。
