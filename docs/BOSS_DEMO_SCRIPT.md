# Boss Demo Script｜老板演示脚本

## 1. 演示目标

本演示不是为了展示“炫技功能”，而是证明 ManJuFlow 可以成为编剧行业 AI 文字内容创作工作台。

演示重点：

- 灵感到剧本；
- 已有剧本到结构化切分；
- 切分结果进入分镜；
- 分镜进入绘图 / 媒体提示词；
- 结果复制 / 导出；
- 后续 AI Assistant 将让非技术人员通过自然语言参与创作；
- 当前暂不强调真实文生图 / 文生视频。

演示时应把产品价值说清楚：ManJuFlow 先解决编剧、短剧策划、漫剧内容团队在“文字内容生产”和“提示词交付”上的效率问题，再考虑接入真实图像、视频和私有生成链路。

## 4. 开场话术

可以直接对老板这样说明：

> 这个项目当前不是为了马上做自动成片，而是先解决编剧和短剧内容团队最刚需的部分：如何把灵感、已有剧本快速转成结构化剧本、分镜和可交付给绘图 / 视频生成环节的提示词。

## 5. 演示路径 A：灵感到剧本到提示词

适用场景：从 0 到 1 的原创内容开发。

演示步骤：

1. 打开网页；
2. 选择“灵感创作”入口；
3. 输入一个安全虚构灵感；
4. 生成结构化剧本；
5. 一键带入分镜；
6. 一键带入绘图 Prompt；
7. 展示复制 / 导出 JSON；
8. 说明这条链路适合原创短剧、漫剧概念开发和内容策划初稿。

安全虚构示例灵感：

```text
一位年轻编剧在深夜地铁站捡到一本没有署名的旧剧本。每当她改写其中一幕，现实中的某个陌生人就会收到一段改变命运的消息。她必须在天亮前找出剧本真正的主人。
```

讲解要点：

- 灵感可以很粗糙；
- 系统会把内容变成结构化剧本；
- 结构化结果可以继续进入分镜和 Prompt；
- 这不是单次聊天，而是可继续传递的工作流。

## 6. 演示路径 B：已有剧本切分到提示词

适用场景：已有剧本二次开发、漫剧化、AI 影视化拆解。

演示步骤：

1. 进入“已有剧本” workspace；
2. 粘贴一段安全虚构短剧文本；
3. 点击“切分已有剧本”；
4. 展示 segments；
5. 带入分镜；
6. 带入绘图 Prompt；
7. 展示导出结果；
8. 说明这条链路适合把已有文字稿拆成可生产的分镜和媒体提示词。

安全虚构短剧文本：

```text
第1集 第1场｜旧电影院｜夜。
雨声敲在破旧的玻璃门上。林棠抱着一只装满手写稿的纸箱，站在空荡的售票厅里。墙上的老海报已经褪色，只剩下一句模糊的标语：故事不会自己消失。

周砚从放映室走出来，手里拿着一卷老胶片。
周砚：你确定要把这些稿子都烧掉？
林棠：它们毁了我父亲，也毁了这个影院。
周砚：也可能，它们是唯一能证明真相的东西。

灯光突然熄灭，放映机自己启动，银幕上出现了林棠父亲年轻时的影像。
```

讲解要点：

- 这条链路适合已有剧本导入；
- 切分结果包含片段摘要、人物、地点、冲突、情绪和视觉备注；
- 切分结果可继续带入分镜；
- 分镜可继续生成绘图 / 媒体提示词。

## 7. 演示路径 C：上传 mock 剧本文档

必须说明：

## 8. 演示路径 D：AI Assistant 规划说明

Assistant 还在规划 / mock 落地阶段，不要夸大为已完成能力。

未来价值：

- 编剧可以自然语言要求 AI 改写；
- 增强钩子；
- 拆分场景；
- 补充人物动机；
- 带入分镜；
- 生成提示词；
- 切换工作区；
- 生成 suggested actions。

边界：

- Assistant 会独立 schema / service / endpoint / prompt；
- 即使使用 DeepSeek，也会与剧本生成、分镜生成、Prompt 生成链路隔离；
- 所有 suggested actions 都需要用户确认；
- Assistant 不应自动修改生产数据。

## 9. 老板关心问题回答

### Q：现在能不能直接做市场？

A：可以进入受控小范围试用 / 演示验证，但还不是生产级 SaaS。当前适合用来验证编剧行业对“AI 文字内容创作工作台”的需求和工作流反馈。

### Q：为什么不先做文生图 / 文生视频？

A：当前老板诉求是编剧行业文字内容创作赋能。先把文字链路、剧本切分、分镜、提示词、Assistant 做稳，后续接生成模型会更稳，也更容易控制成本、安全和交付质量。

### Q：用户上传剧本放哪里？

A：当前 mock 阶段不保存真实文件；真实上传第一版会使用受控本地目录并进入 `.gitignore`，后续用 SQLite 记录 metadata。市场试用前会明确清理策略和隐私提示。

### Q：AI 聊天会不会和剧本生成混在一起？

A：不会。Assistant 会独立 schema / service / endpoint / prompt / env 配置，即使使用 DeepSeek，也与创作链路隔离。后续用 UsageLedger 区分 Assistant Chat 和内容生产链路。

### Q：公开仓库安全吗？

A：公开仓库只放工程主干、mock 能力、文档和安全样例，不放 API Key、真实客户数据、真实服务器、模型权重、私有 workflow。

### Q：还有什么没完成？

A：Assistant UI、Assistant DeepSeek 独立接入、真实上传、聊天记录策略、基础门禁、市场试用总结等仍需继续推进。

## 11. 演示后的下一步建议

老板看完后，可以给出的下一步方向：

- 选择 1～3 个安全虚构样例做产品演示；
- 找 1～2 位编剧进行小范围试用；
- 收集他们对剧本切分、分镜、提示词的反馈；
- 优先落地 AI Assistant；
- 再落地真实 `.docx` 上传；
- 再考虑持久化、登录、部署和商业化试点。

建议把试用反馈聚焦在以下问题：

- 编剧是否愿意从“已有剧本切分”入口开始使用；
- 分镜字段是否符合实际创作习惯；
- Prompt 输出是否方便交给美术或生成工具；
- Assistant 最需要支持哪些 suggested actions；
- 复制 / 导出格式是否足够交接。
