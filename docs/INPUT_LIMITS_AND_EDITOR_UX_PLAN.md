# Input Limits and Editor UX Plan｜输入上限与编辑体验方案

## 1. 文档目的

本文档用于定义 ManJuFlow 市场试用版中所有核心输入框、编辑区和上传提取文本区的字数上限、提示方式、前后端校验策略。

目标是让编剧用户在长时间输入、在线编辑、复制粘贴、上传 Word 文档和后续 AI 处理时，都能提前知道输入边界，避免“写完才发现超限”的糟糕体验。

本步骤只做设计，不修改代码、不接真实 LLM、不引入新依赖。

## 2. 为什么这是 P0 体验问题

这是 P0 体验问题，原因如下：

- 编剧用户会长时间编辑；
- 用户可能一次粘贴大量已有剧本；
- 不能让用户写完才发现超限；
- 必须实时显示当前字数 / 上限；
- 接近上限时要提前提醒；
- 超限时要禁用生成按钮并给出可理解提示；
- 后端也必须校验，防止绕过前端；
- 长文本进入 LLM 前必须分块，不能把 UI 上限误认为单次模型调用上限。

如果输入限制设计不清楚，市场试用时会直接影响用户信任和演示稳定性。

## 3. 输入区分类

核心输入区至少包括：

- 灵感输入 `idea_text`；
- 剧本生成 `extra_requirements`；
- 已有剧本粘贴 `script_text`；
- mock upload `extracted_text`；
- 分镜生成输入；
- ImagePrompt 生成输入；
- Assistant 用户消息；
- Assistant `selected_text`；
- 在线编辑区；
- Word 上传后的 `extracted_text`。

后续如果新增角色设定、世界观设定、Prompt 编辑器、资产备注等输入区，也应复用同一套输入限制和实时计数体验。

## 4. 建议字符上限

市场试用第一版建议：

| 输入区 | 建议上限 | 说明 |
| --- | ---: | --- |
| `idea_text` | 5,000 字符 | 灵感输入通常较短，但允许较完整的故事概念 |
| `extra_requirements` | 2,000 字符 | 用于风格、约束、平台、受众等补充说明 |
| existing script `script_text` | 100,000 字符 | 已有剧本粘贴上限 |
| upload `extracted_text` | 100,000 字符 | Word 提取文本上限 |
| storyboard input | 100,000 字符 | 剧本 / 切分结果进入分镜的文本上限 |
| image prompt input | 100,000 字符 | 分镜文本 / JSON 进入 Prompt 的上限 |
| assistant `user_message` | 4,000～8,000 字符 | 市场试用第一版建议先保守 |
| assistant `selected_text` | 20,000 字符 | 用户选中文本可较长，但不应无限制 |
| online editor draft | 100,000 字符 | 在线编辑稿件上限 |
| 单次 LLM 实际输入 | 后续必须 chunking | 不能等同于 UI 输入上限 |

这些数值是市场试用第一版建议。后续可根据模型上下文、真实用户反馈、生成质量、响应速度和成本继续调整。

## 5. 前端 UX 标准

每个核心输入框应显示：

- 当前字数；
- 最大上限；
- 剩余字数；
- 接近上限提醒；
- 超限状态；
- 超限时禁用生成按钮；
- 超限时给出具体建议。

建议展示文案：

```text
当前 12,340 / 100,000 字
剩余 87,660 字
```

接近上限时：

```text
文本较长，后续生成可能需要更久。建议确认内容已整理清楚。
```

超限时：

```text
已超出 320 字，请删减或拆分后再生成。
```

按钮规则：

- 未超限：允许提交；
- 接近上限：允许提交，但显示提示；
- 已超限：禁用生成按钮；
- 空内容：禁用或给出必填提示；
- mock 上传后自动填入的 `extracted_text` 也必须触发同样计数。

## 6. 后端校验策略

后端不能只相信前端。

后端应：

- 使用统一 limit helper；
- 对 `script_text`、`idea_text`、Assistant `user_message` 等字段校验；
- 对上传后的 `extracted_text` 校验；
- 返回结构化错误；
- 错误信息要适合前端展示；
- 不能把完整用户剧本打进日志；
- 对真实 LLM 模式也要做限制；
- 保持边界测试覆盖。

当前已有基础限制工具：

```text
apps/api/app/services/script_input_limits.py
```

现有常量包括：

- `MAX_SCRIPT_TEXT_CHARS = 100_000`
- `MAX_EXTRACTED_TEXT_CHARS = 100_000`
- `MAX_SCRIPT_UPLOAD_FILE_SIZE_BYTES = 10 * 1024 * 1024`
- `DEFAULT_SCRIPT_CHUNK_SIZE_CHARS = 8_000`
- `DEFAULT_SCRIPT_CHUNK_OVERLAP_CHARS = 400`

后续应把更多输入类型的限制集中管理，避免前端、后端、上传、Assistant、真实 LLM chunking 各自硬编码。

## 7. LLM 模式与 Chunking 边界

必须强调：

- 100,000 字符是 UI / 输入保存上限；
- 100,000 字符不是单次 LLM 调用上限；
- 真实 LLM 调用需要 chunking；
- 长剧本应先切分，再逐段进入模型；
- 市场试用时必须提示用户“长文本会分段处理”；
- chunk 级调用需要进入 UsageLedger；
- 日志不应打印完整 chunk 内容。

建议说明文案：

```text
当前支持最多约 100,000 字符。长文本后续会在后端分段处理，避免超长剧本导致 AI 解析不稳定。
```

## 8. Word 下载 / 上传与编辑关系

后续产品会同时支持：

- 用户在线编辑；
- 下载 Word 离线编辑；
- 再上传继续下一步；
- 上传后提取 `extracted_text`；
- `extracted_text` 进入 Script Segmentation；
- 结构化结果继续进入分镜和 Prompt。

因此：

- 上传后的 `extracted_text` 也要进入相同上限检测；
- Word 上传不应绕过后端限制；
- 在线编辑、下载、上传、再编辑要复用同一套限制策略；
- 后续 Document Round-trip 要记录 source_id、version、metadata；
- 真实上传文件不进入 Git；
- 公开仓库不保存真实剧本文件。

## 9. 验收标准

### 前端验收

- 灵感输入显示字数；
- 已有剧本输入显示字数；
- upload `extracted_text` 显示字数；
- Assistant 输入显示字数；
- 接近上限时有提醒；
- 超限时按钮禁用；
- 错误提示清晰；
- 超限提示能说明需要删减多少字；
- 复制 / 导出不受计数 UI 影响。

### 后端验收

- 超限输入返回 4xx；
- 不超限输入正常；
- 测试覆盖边界值；
- 不打印完整敏感文本；
- 真实 LLM 模式同样执行限制；
- 上传提取文本同样执行限制。

## 10. 后续落地步骤

建议后续小闭环：

- 第 173 步：前端核心输入框字数计数组件；
- 第 174 步：灵感输入和已有剧本输入接入计数；
- 第 175 步：后端统一 limit 错误响应；
- 第 176 步：Assistant 输入限制；
- 第 177 步：在线编辑区限制；
- 第 178 步：Word 上传 `extracted_text` 限制；
- 第 179 步：浏览器验收。

落地原则：

- 先接入最影响演示的输入区；
- 不一次性大改所有 workspace；
- 每一步都跑前端 build；
- 涉及后端校验时补充边界测试；
- 保持错误文案面向非技术用户。
