# ManJuFlow 合作技术资产与权属边界说明草案

## 1. 文档目的

本文档用于合作沟通、技术评审和后续律师起草正式合同前的技术边界说明，不是正式法律合同。

本文档旨在初步明确以下事项之间的边界：

- 公开仓库；
- 合作前既有技术成果；
- 后续合作定制；
- 私有部署；
- 商业化使用。

最终合作条款、授权范围、权属安排和商业使用边界，应以双方正式书面协议为准，并建议由律师审核。

## 2. 项目当前状态

ManJuFlow 当前为公开可评审的 MVP 原型项目。

当前已完成前三阶段：

- Idea → Script；
- Script → Storyboard；
- Storyboard → ImagePrompt。

当前项目已具备：

- FastAPI 后端；
- React / Vite 前端；
- Schema / Prompt / Service / Endpoint / Frontend / Tests / Docs 分层；
- 多模型 provider；
- 请求级模型选择；
- 前端模型选择器；
- 测试与文档体系。

文生图、文生视频、配音、资产管理、任务队列和生产部署等能力尚未作为正式生产能力完成。

## 3. 合作前既有技术资产 Background IP

以下内容可视为合作前既有技术资产：

- 当前 ManJuFlow 仓库代码；
- `README.md` 和 `docs/` 文档；
- Pydantic Schema；
- Prompt 模板；
- Service / parser / LLMClient；
- FastAPI endpoint；
- React 前端页面；
- 后端测试；
- 模型对比样本；
- 本地开发脚本；
- 项目架构与阶段路线。

原则上，上述合作前既有技术资产由项目维护者保留所有权，除非双方另有书面协议。

公开仓库可用于技术评审、项目展示和合作沟通，但公开不等于授予商业使用、再分发、转授权或生产部署权。

## 4. 合作后新增成果 Foreground IP

如果后续与合作公司共同开发，以下新增成果的归属和使用权应在正式合作协议中另行约定：

- 公司定制模块；
- 私有部署适配；
- 工作流定制；
- 业务流程定制；
- 客户项目模板；
- 数据资产管理；
- 生产环境运维脚本；
- 新增前端页面或管理后台；
- 新增模型适配层；
- 新增任务系统或资产系统。

建议在正式协议中明确：

- 哪些成果属于项目维护者；
- 哪些成果属于合作方；
- 哪些成果双方共享；
- 是否允许复用到其他项目；
- 是否允许二次开发、转授权或商业销售。

## 5. 合作方业务资产 Company Assets

合作方业务资产应与公开仓库隔离。

以下内容应归合作方或按正式协议约定处理，不应进入公开仓库：

- 客户数据；
- 剧本素材；
- 商业渠道；
- 公司内部流程；
- 私有 API Key；
- 私有服务器；
- 私有模型配置；
- 客户项目结果；
- 内部运营数据；
- 未公开商业合同信息。

如果后续需要在私有部署环境中使用上述资产，应通过私有仓库、私有服务器、环境变量、密钥管理和权限控制进行隔离。

## 6. 公开仓库边界

公开仓库用于：

- 技术评审；
- Demo 演示；
- 合作沟通；
- 本地运行；
- 架构评估；
- 投资或技术尽调前的非敏感材料展示。

公开仓库不应包含：

- API Key；
- `.env`；
- 真实客户数据；
- 合作方敏感信息；
- 私有服务器地址；
- 模型权重；
- 私有 ComfyUI workflow；
- 未公开商业合同信息；
- 个人隐私信息。

## 7. 第四阶段及以后建议边界

第四阶段及以后如果进入文生图、远端 GPU、ComfyUI、文生视频、资产管理或生产部署，建议继续保持公开仓库和私有部署环境分离。

公开仓库可保留：

- `ImageGenerationInput` / `ImageGenerationOutput` Schema；
- mock endpoint；
- ComfyUI adapter 抽象；
- 环境变量占位；
- 文档和 runbook；
- 本地可运行 demo；
- 不含密钥的测试样例。

私有部署环境应保存：

- 真实 GPU 服务器地址；
- API Key；
- 模型权重；
- 私有 ComfyUI workflow；
- 客户数据；
- 生产任务结果；
- 内部业务流程；
- 私有配置和运维脚本。

## 8. 推荐合作模式

可讨论的合作模式包括：

- 授权使用 + 定制开发；
- 技术服务 + 私有部署；
- 项目公司 / 技术入股；
- 商业授权；
- 联合产品化。

当前不建议在未签署正式协议前，直接整体转让既有项目资产。

具体授权范围、独占性、收益分配、二次开发权、转授权权利和退出机制，应在正式合作协议中明确。

## 9. 需要律师确认的事项

正式合作前建议由律师确认以下事项：

- 既有代码所有权；
- 使用许可范围；
- 是否独占；
- 是否允许转授权；
- 二次开发成果归属；
- 收益分配；
- 保密条款；
- 数据安全；
- 代码托管；
- 终止合作后的使用权；
- 争议解决；
- 公开仓库与私有部署成果的边界；
- 合作方业务资产和客户数据的权属。

## 10. 声明

本文档为技术和商务沟通草案，不构成正式法律意见或最终合同。

任何商业使用、再分发、转授权、生产部署、资产转让或排他性合作安排，均应以双方正式书面协议为准。
